01
URL Parsing
The browser splits the URL into scheme (https), host (google.com), path, query string, and fragment. Each part is validated and canonicalised before anything else happens.
›
02
HSTS Preload Check
Before DNS, the browser checks its built-in HSTS preload list. If the domain is on it, HTTP is silently upgraded to HTTPS — no redirect round trip, no network call wasted.
›
03
DNS Cache Lookup
The browser maintains its own DNS cache separate from the OS. A hit here skips the entire resolution chain — no resolver, no root server, no TLD server involved.
›
04
HTTP Cache Check
If a fresh response is cached (memory or disk), it's returned immediately. Cache-Control, ETag, and Last-Modified headers govern freshness. A cache hit = zero network bytes.
›
05
Service Worker Intercept
If a Service Worker is registered for this origin, it receives a fetch event and can respond from cache, the network, or custom logic — before the browser makes any request.

03 / 08 Physical Journey

WiFi → ISP →
Backbone

The packet physically travels across the planet. Radio waves, fiber optics, and BGP routing work together across milliseconds.

900+
Internet Exchange Points
~4ms
WiFi round trip
packet path · animated
📡
802.11 WiFi
Your device broadcasts on a licensed channel (2.4 GHz or 5 GHz). The access point ACKs every frame. OFDM modulation encodes data in orthogonal subcarriers to resist multipath interference.
›
NAT
NAT — Your Hidden Address
Your router replaces your private 192.168.x.x source address with its public IP before forwarding. Return traffic is mapped back via a connection table. The public internet never sees your device's real address.
›
BGP
BGP — Border Gateway Protocol
~70,000 autonomous systems advertise their IP prefixes to each other via BGP. Your packet follows the best-path announced routes, potentially traversing multiple countries in under 30ms.

04 / 08 TCP

The reliable
pipe

IP delivers packets with no guarantees. TCP adds sequencing, acknowledgement, and retransmission — paying a round-trip handshake as the entry fee.

1 RTT
Handshake cost
3-way
SYN · SYN-ACK · ACK
▶ step through the tcp handshake
SEQ
Sequence Numbers & ACKs
Every byte sent has a sequence number. The receiver sends ACKs confirming delivery. Gaps trigger retransmission. This guarantees ordered, loss-free delivery regardless of underlying IP behaviour.
›
CC
Congestion Control
TCP starts slow, increases the window exponentially (slow start), then backs off on packet loss (AIMD). This cooperative mechanism prevents any single connection from overwhelming the network.

05 / 08 TLS

Locking
the door

Certificates prove identity. ECDHE establishes a shared secret without transmitting it. TLS 1.3 does all this in a single round trip.

1 RTT
TLS 1.3 handshake
90%+
Web now HTTPS
$0
Let's Encrypt certs
certificate chain · key exchange
X.5
X.509 Certificate Chain
The server presents a certificate signed by an intermediate CA, which is signed by a root CA baked into your OS. Your browser walks this chain verifying each signature until it reaches a trusted root.
›
LE
Let's Encrypt & ACME
Let's Encrypt automated certificate issuance via the ACME protocol. HTTP-01 challenge: the CA gives you a token; you serve it at /.well-known/acme-challenge/. Certs are valid 90 days and auto-renew.
›
EC
ECDHE Key Exchange
Both sides pick random private keys, compute public keys on an elliptic curve, and exchange them. Each can compute the same shared secret — but an eavesdropper who captures both public keys cannot. This is forward secrecy.
›
0RT
0-RTT Session Resumption
On reconnection to a known server, TLS 1.3 allows sending encrypted data in the very first message — zero round-trip cost. A session ticket from the previous connection seeds the key derivation.
›
06 / 08 HTTP

The language
of the web

30 years of evolution: text-based HTTP/1.1 → multiplexed HTTP/2 → UDP-based HTTP/3. Each version solved a specific bottleneck.

HTTP/3
Current latest
QUIC
UDP-based transport
▶ compare http versions
Text-based protocol. One request per connection. Persistent connections added in 1.1, but head-of-line blocking means request 2 waits for request 1 to complete. Browsers opened 6 parallel TCP connections per domain as a workaround. Headers sent uncompressed, same cookies repeated every request.
Binary framing + stream multiplexing. Many concurrent requests over a single TCP connection — no queuing. HPACK compresses headers using a static table and Huffman encoding. Server Push lets the server proactively send assets. Widely adopted since 2015.
QUIC replaces TCP entirely. Built on UDP. Connection IDs survive IP changes — switching from WiFi to 4G keeps the session alive. Each stream is independent so one lost packet doesn't stall others. 0-RTT reconnects. Encrypted by default at the transport layer.
HOL
Head-of-Line Blocking Explained
HTTP/1.1: one outstanding request per TCP connection means slow request 1 blocks all others. HTTP/2 solved this at the HTTP layer. HTTP/3 solved the remaining TCP-level version by moving to QUIC/UDP where streams are truly independent.

07 / 08 Response & Cache

Bytes arrive,
cache remembers

The server streams. The browser parses before the file finishes downloading. Smart caching means most bytes are never requested twice.

~26%
Brotli vs gzip saving
304
Not Modified = no body
streaming DOM construction
▶
Streaming Response
The browser doesn't wait for the full HTML. As bytes arrive, the parser builds the DOM incrementally. Sub-resource requests (CSS, JS, images) are dispatched the moment their tags are parsed — the preload scanner runs ahead.
›
BR
Brotli vs Gzip
Brotli uses a static dictionary of 120,000+ common web strings (HTML tags, JS keywords, CSS properties). This gives it a 15–26% size advantage over gzip on text assets. Check Content-Encoding: br in DevTools.
›
CC
Cache-Control Deep Dive
max-age=31536000, immutable — cache forever, use for content-hashed assets. no-cache — always revalidate with ETag before using. no-store — never write to cache, ever. stale-while-revalidate — serve stale instantly, refresh in background.
›
SW
Service Worker Cache
A Service Worker intercepts every fetch from a separate thread. Implement cache-first, network-first, stale-while-revalidate, or any custom strategy. This gives web apps full offline capability.
›
08 / 08 Browser Renders

Bytes become
pixels

Five pipeline stages from HTML tokenisation to GPU compositing. Every frame must complete in 16ms. The pipeline explains why some CSS is free and other CSS is expensive.

16ms
Budget per frame at 60fps
5
Pipeline stages
01
Parse HTML → DOM
Bytes → tokens → nodes. The HTML parser is incremental and fault-tolerant. The preload scanner runs ahead discovering CSS, JS, and images to fetch in parallel with parsing.
~5ms
02
Parse CSS → CSSOM
CSS is render-blocking — the browser cannot paint until all CSS is parsed. The CSSOM is a separate tree mapping selectors to their computed style values.
~2ms
03
DOM + CSSOM → Render Tree
DOM and CSSOM are merged. display:none nodes are excluded. Each visible node gets its computed styles attached. This is the render tree — only what's actually visible.
~1ms
04
Layout — exact geometry
The browser calculates exact pixel coordinates for every element. Changing width, height, margin, or font-size triggers a full reflow. LayoutNG is Chrome's modern constraint-based layout engine.
~3ms
05
Paint → Composite → GPU
Layout → display list of draw commands → promoted layers composited on the GPU thread. transform and opacity animate on the compositor, never triggering layout or paint. That's why they're "free."
~2ms
